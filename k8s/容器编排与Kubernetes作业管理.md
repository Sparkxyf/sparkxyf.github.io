# 容器编排与Kubernetes作业管理（一）pod

## pod

> Pod，是 Kubernetes 项目中最小的 API 对象。如果换一个更专业的说法，我们可以这样描述：Pod，是 Kubernetes 项目的原子调度单位。
>
> pod只是一个逻辑概念。Pod，其实是一组共享了某些资源的容器

==容器的本质是进程==

Kubernetes 所做的，其实就是将“进程组”的概念映射到了容器技术中。

容器的“单进程模型”，并不是指容器里只能运行“一个”进程，而是指容器没有管理多个进程的能力。这是因为容器里 PID=1 的进程就是应用本身，其他的进程都是这个 PID=1 进程的子进程。

> 在 Kubernetes 项目里，Pod 的实现需要使用一个中间容器，这个容器叫作 Infra 容器。在这个 Pod 中，Infra 容器永远都是第一个被创建的容器，而其他用户定义的容器，则通过 Join Network Namespace 的方式，与 Infra 容器关联在一起。

* 容器设计模式：用pod的思想来指导容器编排
* 这个所谓的“组合”操作，正是容器设计模式里最常用的一种模式，它的名字叫：sidecar。顾名思义，sidecar 指的就是我们可以在一个 Pod 中，启动一个辅助容器，来完成一些独立于主进程（主容器）之外的工作。

举个例子：

* war与web服务器
  * 一种方法是，把 WAR 包直接放在 Tomcat 镜像的 webapps 目录下，做成一个新的镜像运行起来。可是，这时候，如果你要更新 WAR 包的内容，或者要升级 Tomcat 镜像，就要重新制作一个新的发布镜像，非常麻烦。
  * 另一种方法是，你压根儿不管 WAR 包，永远只发布一个 Tomcat 容器。不过，这个容器的 webapps 目录，就必须声明一个 hostPath 类型的 Volume，从而把宿主机上的 WAR 包挂载进 Tomcat 容器当中运行起来。
  * 有了 Pod 之后，这样的问题就很容易解决了。我们可以把 WAR 包和 Tomcat 分别做成镜像，然后把它们作为一个 Pod 里的两个容器“组合”在一起
* Istio 项目使用 sidecar 容器完成微服务治理的原理

## 容器设计模式

###单容器的管理模式

在“向上”的方向上，容器可以提供丰富的应用信息，包括特定于应用程序的监控指标（QPS、应用程序健康等），分析开发者感兴趣的信息（线程、堆栈、锁争用、网络消息，sage 统计信息等）、组件配置信息和组件日志。

在“向下”方向，容器界面提供了一个自然的地方来定义一个生命周期，使更容易编写受控制的软件组件管理系统。

###单节点、多容器应用模式

#### Sidecar 模式

1. 首先，容器是资源核算和分配的单位Web 服务器容器的 cgroup可以配置为提供一致的低对查询的延迟响应，而 logsaver 容器配置为在以下情况下清除空闲 CPU 周期Web 服务器不忙。
2. 容器为单位因此将服务和日志保存分开不同的容器可以很容易地划分责任他们在两个独立的程序之间发展的能力并允许他们独立测试。
3. 容器是单位重用，所以Sidecar容器可以与配对大量不同的“主”容器（例如日志保护程序配置）。
4. 容器提供了一个故障容器——边界，使整个系统成为可能优雅地降级。

####Ambassador模式

大使容器代理通信往返于主容器。应用程序认为它只是在与单个内存缓存交互，但实际上是跨多个内存的分布式安装的任务。

####适配器模式

最终单节点模式是适配器模式。与大使模式相比，它提供了一个具有简化视图的应用程序外部世界，适配器呈现外部世界，以简化的、同质化的应用视图。他们通过标准化输出和接口来做到这一点跨多个容器。一个具体的例子，适配器模式是确保所有容器在一个系统具有相同的监控界面。

### 多节点应用模式

除了在单个机器上协作容器之外，模块化容器使构建协调的多节点分布式应用程序变得更加容易。接下来我们将描述其中三种分布式系统模式。与上一节中的模式一样，这些也需要系统支持 Pod 抽象。

#### Leader选举模式

分布式系统中最常见的问题之一是领导者选举。虽然复制通常用于在一个组件的多个相同实例之间共享负载，但另一种更复杂的复制用途是在需要将一个副本与作为“领导者”的集合区分开来的应用程序中。如果领导者失败，其他副本可以快速取代领导者。将领导选举库链接到应用程序的另一种方法是使用领导选举容器。一组领导选举容器，每个容器都与一个需要领导选举的应用程序实例共同调度，可以在它们之间执行选举，并且它们可以通过本地主机向每个需要领导选举的应用程序容器提供一个简化的 HTTP API（例如成为领导者、更新领导权等）。这些领导选举容器可以由这个复杂领域的专家构建一次，然后应用程序开发人员可以重新使用随后的简化接口，而不管他们选择哪种实现语言。这代表了软件工程中最好的抽象和封装。

####工作队列模式

开发者只需要构建一个容器，可以将文件系统上的输入数据文件转换为输出文件；这个容器将成为工作队列的一个阶段。开发完整工作队列所涉及的所有其他工作都可以由通用工作队列框架处理，只要需要这样的系统，就可以重用该框架。

####分散/聚集模式

要实现分散/收集系统，用户需要提供两个容器。

一、实现叶节点计算的容器；此容器执行部分计算并返回相应的结果。二、第二个容器是合并容器；该容器采用所有叶容器的聚合输出，并将它们分组为单个响应。

很容易看出用户如何通过提供实现这些相对简单接口的容器来实现任意深度的分散/聚集系统。

##总结

你就可以把整个虚拟机想象成为一个 Pod，把这些进程分别做成容器镜像，把有顺序关系的容器，定义为 Init Container。这才是更加合理的、松耦合的容器编排诀窍，也是从传统应用架构，到“微服务架构”最自然的过渡方式。

> 强行把整个应用塞到一个容器里，等价于 Docker In Docker
